# WHO Is Doing Interesting Work In The Layers *BELOW* CUDA?

I decided to *start* curating this list which draws from different dev communities (AMD, NVIDIA, Intel, Khronos, LLVM, Apache, Google, Modular, Linux kernel, Jane Street, academic/open research) to furnish me with some exposure to what happens in the different dev communities that are concerned with what happens BELOW CUDA. 

This **interest in these dev communities** has sort of grown out of my curiosity with ROCm/HIP foundations, LLVM compilers, different software engineering standards, and different eBPF-like tracing tools. It's a really sort of a two-pronged thing ... the topic AND the meta-topic that's about the developers and their workflows ... in order to understand workflows, or even ask kind of basic questions, or communicate with the people in these dev communities, it's necessary to have a somewhat solid working knowledge in the material they are currently working on.

Of course, this is typically viewed as *HARD* close-to-metal stuff. As a rule, exploring these technologies will build the mental model needed to work directly on GPU kernels, runtimes, and compilers for AI at the hardware level. Many link to further sub-repos or docs for deeper dives into the details of the details behind GPU drivers/runtimes, compilers/IRs (LLVM/MLIR), heterogeneous programming models (HIP, SYCL, OpenCL), kernel libraries, tensor compilers, low-level AI accelerators, debugging/profiling/tracing tools, and related infrastructure for working at the metal level in AI/HPC.

1. **ROCm Platform (AMD)** - [https://github.com/ROCm/ROCm](https://github.com/ROCm/ROCm)  
   This open-source stack provides the full alternative runtime and drivers to CUDA for AMD GPUs, exposing HSA/ROCk kernel interfaces and HIP for direct hardware access in AI workloads. It reveals how vendor-specific compute platforms are built from kernel modules up through math libraries.  
   After investigating, pursue individual ROCm component repositories like HIP or rocBLAS for deeper kernel and BLAS implementation details.

2. **HIP (AMD)** - [https://github.com/ROCm/HIP](https://github.com/ROCm/HIP)  
   HIP is AMD's CUDA-like C++ kernel language and runtime that compiles to AMD GPU ISA, allowing portable low-level kernel writing without NVIDIA lock-in. It demonstrates how to target hardware-specific features like wavefronts and shared memory directly.  
   Next, explore HIPIFY tools or port existing CUDA kernels to see migration paths and ISA differences.

3. **MIOpen (AMD)** - [https://github.com/ROCm/MIOpen](https://github.com/ROCm/MIOpen)  
   MIOpen is AMD's open-source deep learning primitives library (convolutions, GEMM, etc.) optimized at the kernel level for ROCm GPUs. It shows how AI-specific operations are implemented and tuned below framework abstractions.  
   After this, investigate Composable Kernel for even lower-level, composable GEMM and fusion primitives.

4. **Composable Kernel (AMD)** - [https://github.com/ROCm/composable_kernel](https://github.com/ROCm/composable_kernel)  
   This library provides highly optimized, template-based kernels for matrix operations and fusions on AMD GPUs, exposing tile-level and wavefront-level control. It illustrates modern techniques for achieving peak hardware utilization in AI training/inference.  
   Next, study rocBLAS/hipBLAS to see how these kernels integrate into higher BLAS APIs.

5. **rocBLAS (AMD)** - [https://github.com/ROCm/rocBLAS](https://github.com/ROCm/rocBLAS)  
   rocBLAS delivers BLAS operations optimized for AMD GPUs using HIP and low-level kernels. It reveals vendor-specific tuning for linear algebra critical to AI.  
   After exploring, move to rocSOLVER or rocFFT for solver and FFT primitives at the same layer.

6. **rocProfiler (AMD)** - [https://github.com/ROCm/rocprofiler](https://github.com/ROCm/rocprofiler)  
   rocProfiler enables low-level performance counter collection and tracing on AMD GPUs, analogous to NVIDIA CUPTI. It teaches how to instrument kernels and runtimes for hardware-level bottlenecks.  
   Next, examine Omniperf/Omnitrace for higher-level analysis and causal profiling tools.

7. **Omniperf (AMD)** - [https://github.com/AMDResearch/omniperf](https://github.com/AMDResearch/omniperf)  
   Omniperf is a kernel-level profiler with GUI for AMD Instinct GPUs, exposing counters for memory hierarchy and compute units. It directly aids understanding of AI kernel performance at the metal.  
   After this, explore Linux amdgpu driver sources for even deeper hardware interaction.

8. **NVIDIA Open GPU Kernel Modules** - [https://github.com/NVIDIA/open-gpu-kernel-modules](https://github.com/NVIDIA/open-gpu-kernel-modules)  
   These are NVIDIA's open-source Linux kernel drivers for Turing+ GPUs, exposing the lowest-level hardware interface. They demystify how proprietary CUDA ultimately rests on open kernel code.  
   Next, investigate Nouveau driver for community reverse-engineered NVIDIA support.

9. **CUTLASS (NVIDIA)** - [https://github.com/NVIDIA/cutlass](https://github.com/NVIDIA/cutlass)  
   CUTLASS provides CUDA C++ templates for high-performance GEMM and epilogue fusions at the warp and thread-block level. It exemplifies expert-level kernel engineering below cuBLAS.  
   After studying, dive into Triton for higher-productivity kernel writing that still targets similar hardware primitives.

10. **Triton (OpenAI/triton-lang)** - [https://github.com/triton-lang/triton](https://github.com/triton-lang/triton)  
    Triton is a Python-embedded DSL and compiler for writing efficient GPU kernels without manual CUDA C++. It bridges productivity and low-level control for AI custom ops.  
    Next, explore TVM or MLIR backends that Triton can target for broader hardware portability.

11. **Apache TVM** - [https://github.com/apache/tvm](https://github.com/apache/tvm)  
    TVM is an end-to-end compiler stack that lowers models to optimized kernels across CPUs, GPUs, and accelerators via its tensor expression language. It shows automated code generation at the metal level.  
    After this, study its integration with MLIR or Halide for advanced IR transformations.

12. **LLVM Project** - [https://github.com/llvm/llvm-project](https://github.com/llvm/llvm-project)  
    LLVM provides the modular compiler infrastructure (including Clang and backend codegen) that underpins CUDA, HIP, and many AI compilers. It is foundational for lowering to GPU ISAs like PTX or AMDGCN.  
    Next, focus specifically on the MLIR subproject for multi-level IR tailored to AI accelerators.

13. **MLIR (LLVM)** - [https://mlir.llvm.org/](https://mlir.llvm.org/) (core in llvm-project)  
    MLIR offers extensible dialects and transformations for hardware-specific AI compilation, used in XLA, IREE, and Triton. It enables reusable lowering from high-level ops to device kernels.  
    After exploring, examine Torch-MLIR or IREE for framework-specific frontends.

14. **Torch-MLIR** - [https://github.com/llvm/torch-mlir](https://github.com/llvm/torch-mlir)  
    Torch-MLIR bridges PyTorch models into the MLIR ecosystem for compilation to GPUs and accelerators. It reveals how framework graphs are lowered to hardware-native code.  
    Next, investigate IREE or OpenXLA for full runtime execution environments built on MLIR.

15. **IREE (Google)** - [https://github.com/iree-org/iree](https://github.com/iree-org/iree)  
    IREE is an MLIR-based compiler and runtime for deploying ML models to diverse hardware with low-level control over tiling, scheduling, and memory. It demonstrates portable "below CUDA" execution.  
    After this, explore OpenXLA/XLA for Google's production tensor compiler stack.

16. **OpenXLA / XLA** - [https://github.com/openxla/xla](https://github.com/openxla/xla)  
    XLA (Accelerated Linear Algebra) is Google's open compiler for optimizing ML computations on TPUs/GPUs via HLO IR and MLIR. It shows vendor-agnostic low-level fusion and codegen.  
    Next, study oneAPI/SYCL for cross-vendor heterogeneous programming alternatives.

17. **oneAPI Level Zero** - [https://github.com/oneapi-src/level-zero](https://github.com/oneapi-src/level-zero)  
    Level Zero is Intel's low-level runtime API for direct access to accelerators in the oneAPI stack. It provides fine-grained control similar to CUDA driver API but openly standardized.  
    After exploring, examine SYCL implementations for higher-level C++ programming.

18. **SYCL (Khronos/oneAPI)** - [https://github.com/oneapi-src/SYCLomatic](https://github.com/oneapi-src/SYCLomatic) (and samples)  
    SYCL is a Khronos open standard for single-source C++ heterogeneous programming that targets GPUs without vendor lock-in. It abstracts kernels and memory while allowing low-level tuning.  
    Next, look at portBLAS or portDNN for SYCL-based math and DNN primitives.

19. **portBLAS (Codeplay/oneAPI)** - [https://github.com/codeplaysoftware/portBLAS](https://github.com/codeplaysoftware/portBLAS)  
    portBLAS implements BLAS using SYCL for portable GPU execution across vendors. It demonstrates open alternatives to cuBLAS at the kernel level.  
    After this, explore OpenCL as the foundational standard SYCL builds upon.

20. **OpenCL (Khronos)** - [https://github.com/KhronosGroup/OpenCL-ICD-Loader](https://github.com/KhronosGroup/OpenCL-ICD-Loader) (and related specs/repos)  
    OpenCL is the original open standard for heterogeneous compute, with loaders and runtimes exposing device kernels and memory models. It underpins many modern GPU AI stacks.  
    Next, investigate Vulkan Compute shaders or SPIR-V for graphics-oriented low-level compute.

21. **Modular Platform / MAX Kernels** - [https://github.com/modular/modular](https://github.com/modular/modular)  
    Modular's open MAX kernels and Mojo standard library provide high-performance GPU kernels (NVIDIA/AMD/Apple) written in Mojo for AI inference. They illustrate modern "Python-to-metal" programming.  
    After exploring, dive into Mojo language itself for systems-level GPU coding.

22. **Mojo Standard Library (Modular)** - [https://github.com/modular/modular/tree/main/mojo/stdlib](https://github.com/modular/modular/tree/main/mojo/stdlib)  
    Mojo's open stdlib includes low-level GPU programming primitives and kernels optimized for AI hardware. It shows how a modern language can replace CUDA C++ at the metal.  
    Next, study Halide or TileLang for domain-specific kernel languages.

23. **Halide** - [https://github.com/halide/Halide](https://github.com/halide/Halide)  
    Halide is a DSL for image processing and tensor computations that separates algorithm from schedule, generating optimized GPU kernels. It pioneered many low-level AI compiler techniques.  
    Next, explore TileLang or FlashAttention implementations for attention-specific kernels.

24. **TileLang** - [https://github.com/tile-ai/tilelang](https://github.com/tile-ai/tilelang)  
    TileLang is a DSL built on TVM for writing high-performance GPU/accelerator kernels with tiling abstractions. It targets low-level optimizations for AI primitives.  
    After this, investigate FlashAttention repos for memory-efficient transformer kernels.

25. **FlashAttention** - [https://github.com/Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention) (CUDA/Triton variants)  
    FlashAttention implements IO-aware attention kernels at the CUDA/Triton level for transformers, fusing operations to minimize memory traffic. It exemplifies state-of-the-art low-level AI kernel design.  
    Next, explore Linux GPU drivers like amdgpu for hardware scheduling insights.

26. **amdgpu Linux Kernel Driver** - [Linux kernel source (AMD contributions)](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/drivers/gpu/drm/amd)  
    The open amdgpu driver in the Linux kernel handles AMD GPU hardware initialization, scheduling, and compute queues. It is the true metal layer for ROCm.  
    After this, examine Nouveau for NVIDIA community drivers.

27. **Nouveau Driver** - [https://gitlab.freedesktop.org/nouveau](https://gitlab.freedesktop.org/nouveau) (part of Mesa/Linux)  
    Nouveau is the open-source reverse-engineered driver for NVIDIA GPUs in Linux, providing compute support without proprietary modules. It reveals undocumented hardware interfaces.  
    Next, study eBPF-based GPU tracing tools for runtime observability.

28. **Ingero (eBPF GPU tracing)** - [https://github.com/ingero-io/ingero](https://github.com/ingero-io/ingero)  
    Ingero uses eBPF to trace CUDA/ROCm runtime calls and kernel events with causal observability. It enables low-overhead debugging deep in the stack.  
    After exploring, look at Perfetto or Tracy for visualization of traces.

29. **Perfetto** - [https://github.com/google/perfetto](https://github.com/google/perfetto)  
    Perfetto is a high-performance tracing system that can capture GPU/CPU events across heterogeneous systems. It is used for low-level performance analysis in AI runtimes.  
    Next, investigate rocTracer or CUPTI alternatives in ROCm/NVIDIA open tools.

30. **rocTracer (AMD)** - [https://github.com/ROCm/roc-tracer](https://github.com/ROCm/roc-tracer) (part of ROCm ecosystem)  
    rocTracer provides API and kernel tracing for HIP/OpenCL on AMD GPUs. It complements profiling for full-stack visibility below CUDA equivalents.
