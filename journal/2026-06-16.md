# Tool-calling reliability engineering of agentic harnesses

This is the single most consistent source of pain when people move from “wow, the model called a tool” demos to anything that has to run repeatedly without a human babysitting it.

Frontier models are still shockingly bad at producing correct, complete, and stable tool calls once you leave the narrow band of well-represented examples in their training data. They hallucinate parameter names, invent values that don’t exist, drop required arguments, use the wrong types, or call tools in sequences that are logically impossible. The model’s “reasoning” can look sophisticated while the actual function call it emits is garbage.

The critical insight is that **this is not primarily a model problem anymore**. The models have gotten better at understanding what you want. The failure rate stays high because the harness layer around the model is usually too thin. A serious harness has to treat every tool call as untrusted output that must be:

- Parsed and validated against a strict schema
- Retried with corrected arguments when it fails
- Sandboxed so a bad call doesn’t destroy state or cost real money
- Logged with enough context to debug why the model chose that call
- Sometimes routed through a secondary verification step or smaller model

Most public “agent” frameworks still treat the model’s tool call as mostly trustworthy and only add light post-processing. That works for Twitter demos. It collapses in production.

This is also where the guardrail excuse gets especially dishonest. When a company’s agent produces dangerous or expensive tool calls, they often say “we need stronger guardrails.” What they usually mean is “our base model is too uncontrolled and we’re papering over it with extra filtering instead of admitting the architecture is fragile.” The people who have actually built robust systems know the guardrails have to live in the harness (schema enforcement, allow-lists, cost caps, human approval gates), not just in the model’s system prompt.

The practical signal here is that **tool-calling reliability is now mostly an engineering and harness problem, not a scaling problem**. The teams getting real work done are spending their effort on deterministic validation layers, typed tool definitions, explicit retry and correction loops, and observability that shows exactly which tool call failed and why — rather than waiting for the next 10× bigger model to magically become more obedient.

This pattern keeps resurfacing because it directly attacks the central delusion: that frontier model capability is the scarce resource. In reality, once you’re past a certain competence threshold, the scarce resource is the harness discipline that turns unreliable model output into reliable system behavior. That’s the melon worth caving in.

## Suggested Strategies

### Elimination of Tool Surface Area

The first strategy centers on radical elimination of tool surface area as the primary reliability lever. In agentic harnesses, every additional tool or parameter dramatically multiplies opportunities for malformed or hallucinated calls that the model cannot reliably avoid. A simplicity-first approach therefore begins by ruthlessly pruning the tool inventory to the smallest set that still achieves the mission, rather than layering validation around an ever-growing surface. Each retained tool must carry an extremely narrow, statically typed interface that removes entire classes of argument errors before any model output occurs. This design choice directly attacks the root cause of tool-calling failures instead of attempting to detect them after the model has already emitted an invalid call. By limiting dynamic tool selection and forbidding runtime generation of new tool signatures, the harness eliminates the failure mode at its origin. The resulting system requires far less retry logic, sandboxing, and observability overhead because fewer things can go wrong. Teams that adopt this stance stop treating model scale as the solution and instead treat tool minimization as the reliability specification. The outcome is a harness whose correctness depends primarily on its restricted scope rather than on the model’s variable obedience. This approach accepts reduced apparent capability in exchange for dramatically higher realized reliability.

### Render Tool Calls Impossible

The second strategy uses structural constraints that render invalid tool calls physically impossible within the harness architecture. Rather than relying on post-emission validation or model instruction, the design enforces capability-based access and static allow-lists at the point where tool definitions are registered. Every tool call is mediated through an immutable capability token that encodes exactly which operations and argument ranges are permitted, so the model never receives the ability to request disallowed actions. This eliminates entire categories of hallucinated parameters and out-of-scope calls by making them inexpressible in the interface the model sees. The harness therefore invests engineering effort in the boundary definition itself instead of building elaborate correction loops downstream. Because invalid calls cannot be formed, the system avoids the compounding complexity of retries, human approval gates, and secondary verification models. The approach treats tool-calling reliability as an access-control problem solved through deliberate restriction rather than through increased monitoring or model improvement. Over time the harness becomes simpler to audit and maintain precisely because its failure space has been deliberately narrowed at design time. This method directly counters the tendency to compensate for model unreliability with additional layers that themselves introduce new failure modes.

### Minimalist But Aggressive Validation Where Flexibility Is Necessary 
 
The third strategy applies aggressive but minimalist validation at the exact moment of tool-call emission while preserving maximum operational flexibility. The harness treats every model-generated call as untrusted by default and subjects it to a small, deterministic schema validator and range checker before any execution occurs. When a call fails validation, the harness returns a precise, structured error to the model along with the corrected schema fragment, enabling immediate self-correction without human intervention or complex planning. This loop is kept deliberately lightweight so that the added reliability mechanism does not itself become a source of latency or new failure modes. The design rejects the assumption that larger models or richer prompts will eventually solve the problem and instead places the reliability burden on fast, local verification that operates independently of model scale. By focusing correction effort exclusively at the emission boundary, the harness avoids the need for heavy sandboxing or extensive logging of downstream effects. The result is a system that maintains broader tool access while still achieving high reliability through rapid, low-complexity feedback rather than through elimination or structural prohibition. This stance recognizes that some operational breadth is necessary and therefore concentrates engineering resources on making the validation step itself extremely reliable and simple.
